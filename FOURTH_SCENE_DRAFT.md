# Fourth Scene Draft v0.2

## 場景定位

- Scene ID: `smart-care-mission-station`
- 場景名稱：`智慧照護任務站`
- 主題：`智慧醫療 + 長照照護`
- 風格：情境敘事更重、人物元素更多、科技更有溫度的照護任務場景
- 玩家身份：智慧照護任務員

## 核心訊息

這個場景要傳達的不是「設備很厲害」，而是：

- 科技如何幫助長者更安全
- 科技如何讓照護者更即時掌握狀況
- 科技如何讓家屬更安心
- 科技如何在不取代陪伴的前提下，成為照護支持的一部分

核心句可定為：

**智慧照護不是把人變成數據，而是用科技讓照護更即時、更細緻，也更有尊嚴。**

## 場景為什麼適合台灣情境

這個主題非常適合台灣當前社會現況，因為它自然對應到：

- 高齡化
- 長照需求增加
- 家屬遠距關心需求
- 跌倒與離床風險
- 生理感測與照護提醒
- 智慧病房 / 智慧照護空間

它也讓整個專案的四個場景形成更完整的應用光譜：

1. `晶片獵人`
   AI / 晶片 / 基礎系統認識
2. `電子工程雙體驗`
   展攤互動 / VR / 無人機 / 訊號整合
3. `智慧工廠任務站`
   自動化 / 感測 / 控制 / 產線邏輯
4. `智慧照護任務站`
   醫療照護 / 長照支持 / 以人為核心的科技應用

## 設計原則

這一個場景會比前三個更細緻，調整方向如下：

- 題數增加到 `10 題`
- 題目敘述更長
- 每題盡量有角色、情境與照護目的
- 不做成冷知識測驗
- 強調科技「協助照護」，不是「取代照護」

## 場景故事

玩家進入一個結合智慧醫療與長照支持的照護中心，協助觀察多位長者的狀況。

在這裡，系統會出現不同提醒：

- 生命徵象波動
- 離床與跌倒風險
- 用藥提醒
- 睡眠與活動狀態
- 家屬遠距查詢

玩家要協助判斷：

- 先關注什麼
- 哪種裝置最能幫忙
- 哪種流程最合理
- 科技在照護中扮演的角色應該是什麼

## 流程總覽

1. 開場：智慧照護真正想幫助的是什麼
2. 分支：先看生命徵象，或先看環境警示
3. 限時：快速判斷哪個提醒最需要先處理
4. 圖片：辨識智慧照護設備
5. 情境：跌倒 / 離床 / 安全提醒
6. 配對：裝置與功能
7. 分支：照護現場優先順序
8. 排序：異常事件處理流程
9. 情境：科技如何協助但不能取代陪伴
10. 收束：為何智慧醫療與長照也是電子工程的重要舞台

## 完整題目草案

### Q1

- Type: `single-choice`
- 題目：
  `林奶奶剛入住智慧照護中心。照護團隊希望在不頻繁打擾她休息的情況下，仍能更早注意到身體狀況與安全風險。從這個情境來看，智慧照護最核心想幫助的是什麼？`
- 選項：
  - A：讓照護空間看起來更高科技
  - B：讓感測、提醒與照護判斷更即時，幫助安全與陪伴
  - C：讓所有照護工作都完全交給機器
  - D：讓家屬完全不用再關心長者狀況
- 正解：B
- Trait 分布：
  - B -> `care`
  - A -> `systems`
  - C -> `support`
  - D -> `vitals`

## 共通主線

### Q5

- Type: `single-choice`
- 題目：
  `一位長者在照護中心內行走比較不穩，但又不希望被過度打擾。從照護角度來看，跌倒偵測或離床提醒設備最大的價值是什麼？`
- 選項：
  - A：讓照護者能更早注意風險，必要時即時協助
  - B：讓照護空間看起來更像科技展
  - C：讓照護者完全不用再巡房
  - D：讓長者永遠不能自己活動
- 正解：A
- Trait 分布：
  - A -> `care`
  - B -> `systems`
  - C -> `support`
  - D -> `vitals`

### Q6

- Type: `matching`
- 題目：
  `請把智慧照護裝置和最接近的功能配對起來。`
- 建議配對：
  - 血氧 / 生命徵象感測 -> 監測生理狀況
  - 離床提醒 -> 協助注意安全風險
  - 用藥提醒設備 -> 協助規律照護
  - 遠距照護平台 -> 讓家屬或照護團隊掌握狀況
- Trait 主軸：

- 選項：

  - B -> `family-q8`

  - A -> `care`
  - B -> `support`

### Q8

- Type: `ordering`
- 題目：
  `當照護系統發出異常提醒後，哪一個處理順序最合理？`
- 建議排序：
  1. 先確認警示內容
  2. 判斷長者當下狀況
  3. 必要時通知照護者或前往協助
  4. 記錄與回報狀況
  5. 持續追蹤後續變化
- Trait 主軸：
  - `support`

### Q9

- Type: `single-choice`
- 題目：
  `智慧照護系統可以協助提醒、判讀與追蹤，但在長照現場，哪一件事仍然最不能被科技取代？`
- 選項：
  - A：人的陪伴、關心與實際互動
  - B：所有決策都只看儀器數值
  - C：完全不用再和長者說話
  - D：只要有數據就不需要照護經驗
- 正解：A
- Trait 分布：
  - A -> `care`
  - B -> `systems`
  - C -> `support`
  - D -> `vitals`

### Q10

- Type: `single-choice`
- 題目：
  `為什麼智慧醫療與長照照護也屬於電子工程的重要舞台？`
- 選項：
  - A：因為只要出現在醫療場域的設備都一定是 AI
  - B：因為感測、警示、資料傳輸、裝置設計與系統整合都需要電子工程能力
  - C：因為醫療設備只和醫師有關，和工程沒有關係
  - D：因為照護問題只靠軟體畫面就能解決
- 正解：B
- Trait 分布：
  - B -> `systems`
  - A -> `vitals`
  - C -> `support`
  - D -> `care`

## Trait 設計

### Trait 1

- ID: `vitals`
- 代表方向：
  對生命徵象、感測資訊與身體狀況變化很敏感

### Trait 2

- ID: `care`
- 代表方向：
  重視長者安全、安心感與照護品質

### Trait 3

- ID: `systems`
- 代表方向：
  擅長理解裝置、平台、訊號與資料怎麼整合

### Trait 4

- ID: `support`
- 代表方向：
  重視照護流程、現場支援與實際協助

## 結果卡草案

### 1. 生命訊號觀察員

- 對應 Trait：`vitals`
- 副標籤建議：
  - `細膩判讀型`
  - `狀態敏感型`
- 文案方向：
  你很容易注意到生理數據與狀態變化的細節，適合站在第一線幫助早期觀察與判讀風險。

### 2. 智慧照護協調員

- 對應 Trait：`care`
- 副標籤建議：
  - `安心守護型`
  - `關懷整合型`
- 文案方向：
  你最在意的不只是系統有沒有提醒，而是長者有沒有被真正照顧到，是否感受到安全與陪伴。

### 3. 醫療系統整合師

- 對應 Trait：`systems`
- 副標籤建議：
  - `訊號串接型`
  - `平台整合型`
- 文案方向：
  你最容易看見裝置、訊號、資料與照護平台之間的關係，很像把整套智慧照護系統串起來的人。

### 4. 長照科技守護員

- 對應 Trait：`support`
- 副標籤建議：
  - `現場支援型`
  - `流程陪伴型`
- 文案方向：
  你重視的不只是科技功能，而是它能不能真的幫到照護現場、減輕壓力、讓家屬更放心。
